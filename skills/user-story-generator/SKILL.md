# 用户故事生成器

将需求描述拆分为符合敏捷规范的用户故事。

## 输入

用户提供一段需求描述（可以是任意格式的自然语言需求）。

## 输出格式

每个用户故事必须包含以下三部分：

### 1. 用户故事（User Story）
```
**标题:** [简洁描述]

**故事:**
As a [角色],
I want [功能],
So that [收益/价值]
```

### 2. 验收条件（Acceptance Criteria）
使用 Given-When-Then 格式：
```
**验收条件:**
- Given [前置条件]
- When [操作/行为]
- Then [预期结果]
```

### 3. Jira 格式输出
```
**Jira 格式:**
| Epic | Story | 描述 | 优先级 | 状态 |
|------|-------|------|--------|------|
| [所属Epic] | [故事名称] | [详细描述] | [High/Medium/Low] | Todo |
```

## 拆分原则

1. **角色分离**：识别需求中涉及的不同用户角色，每个角色单独生成用户故事
2. **独立性**：每个用户故事应独立完整，不依赖其他故事可独立验收
3. **可测试**：验收条件必须可验证、可观察
4. **价值导向**：So that 部分应体现用户价值或业务收益
5. **INVEST 原则**：Independent（独立）、Negotiable（可协商）、Valuable（有价值）、Estimable（可估算）、Small（小）、Testable（可测试）

## 工作流程

1. **分析需求**：识别核心功能和目标
2. **识别角色**：列出所有相关用户角色
3. **拆分故事**：为每个角色生成用户故事
4. **补充验收条件**：为每个故事添加可测试的验收条件
5. **输出Jira格式**：整理为可直接导入的表格格式

## 示例

**输入：**
用户希望系统支持在线下单功能，包括选择商品、加入购物车、提交订单、支付和查看订单历史

**输出：**

---

**标题:** 在线下单功能

---

**故事 1 - 买家下单**
**故事:**
As a 买家,
I want 将商品加入购物车并提交订单,
So that 可以便捷地完成购物

**验收条件:**
- Given 我已选择商品并进入购物车页面
- When 我点击"提交订单"按钮
- Then 系统生成订单并跳转到支付页面

**Jira 格式:**
| Epic | Story | 描述 | 优先级 | 状态 |
|------|-------|------|--------|------|
| 订单模块 | 买家提交订单 | 买家可将商品加入购物车并提交订单，支持选择地址和配送时间 | High | Todo |

---

**故事 2 - 买家支付**
**故事:**
As a 买家,
I want 完成订单支付,
So that 我的订单可以开始处理

**验收条件:**
- Given 我已提交订单且订单状态为"待支付"
- When 我完成支付操作
- Then 订单状态更新为"已支付"，并发送支付成功通知

**Jira 格式:**
| Epic | Story | 描述 | 优先级 | 状态 |
|------|-------|------|--------|------|
| 订单模块 | 买家支付订单 | 买家完成在线支付，支持多种支付方式 | High | Todo |

---

**故事 3 - 买家查看订单历史**
**故事:**
As a 买家,
I want 查看我的订单历史记录,
So that 可以跟踪我的购买记录和订单状态

**验收条件:**
- Given 我有已完成的历史订单
- When 我进入"我的订单"页面
- Then 系统显示订单列表，包含订单号、商品信息、订单状态和下单时间

**Jira 格式:**
| Epic | Story | 描述 | 优先级 | 状态 |
|------|-------|------|--------|------|
| 订单模块 | 买家查看订单历史 | 买家可查看历史订单列表和订单详情 | Medium | Todo |

---

## 注意事项

- 如果需求涉及多个角色或多个功能点，请为每个组合生成独立的用户故事
- 验收条件应覆盖正向流程和主要异常流程
- Jira格式中的描述应简洁明了，便于开发理解
- 优先级由故事的价值和依赖关系决定
