# QA 需求评审 Skill

## TL;DR

**30 秒了解此 skill**：15 测试维度 × 2 级分类 → 结构化评审报告（MD + HTML）→ 存档到 `docs/`

1. 确认范围 → 2. 逐页面走查（含跨页面比对）→ 3. 生成报告

Quick 模式只查 ✖ 必须修正类问题（5 个核心测试维度），Full 模式走全部 15 维度。

## Overview

以**软件测试总工程师**视角对 PRD 需求文档进行系统性评审，重点识别：

- 无法编写测试用例的模糊描述
- 界面控件规格缺失（类型、长度限制、数据源）
- 测试边界无法确定的条件
- 缺少验收标准导致上线验收风险
- 异常流程和权限场景的测试盲区

**Core principle**：需求阶段发现问题的修复成本，是测试阶段的 1/10，是上线后的 1/100。

## When to Use

**触发条件（满足任一）：**

- 开始编写测试计划或测试用例前
- 接到新版需求文档后
- 测试过程中发现需求描述无法支撑用例设计
- 接手他人测试工作，需要重新梳理需求
- 需求评审会议前，提前准备测试视角的问题清单

**具体症状：**

- 需求文档没有明确的验收标准，不知道测什么算通过
- 界面控件缺少类型、长度等规格，边界值无法确定
- 下拉选项来源不明，无法准备测试数据
- 必填/非必填字段未标注，空值测试无依据
- 新增/编辑功能无唯一性规则说明
- 异常流程和错误提示文案缺失
- 同一功能在不同章节描述矛盾

## When NOT to Use

- 需求文档已通过测试评审会议且用例已编写完成
- 仅有概念性讨论，尚未形成文档
- 紧急 hotfix，无需完整评审
- 极小的 UI 样式调整，不影响功能逻辑

## 执行流程

评审应按以下顺序执行，避免遗漏：

**第零步：确认评审范围 [检查点 0]** — 向用户展示扫描结果并确认，输出格式如下：

```markdown
**[检查点 0] 评审范围确认**

| #   | 页面/模块 | 类型                            | 建议聚焦维度   | 测试复杂度 |
| --- | --------- | ------------------------------- | -------------- | ---------- |
| 1   | [页面名]  | [表单型/列表型/流程型/仪表盘型] | [维度编号]     | [高/中/低] |
| ... |           |                                 |                |            |

**全局视图**：[N] 个页面/模块，核心业务流程为 [一句话描述]，模块间依赖关系 [简述]。

**高风险模块**：[模块名]（原因：[测试角度的风险描述]）

**建议审阅深度**：[Quick / Full] 模式（[理由]）

请确认是否开始 [Quick/Full] 模式评审，或调整为其他模式？
```

**第一步：逐页面走查** — 包含两个子阶段：

**子阶段 1a：全局扫描** — 通读整篇文档，产出全局视图：

- 页面/模块清单及其类型判断
- 核心业务流程（一句话：如"申请→审批→通知→归档"）
- 模块间依赖关系（如"编辑页依赖列表页的数据选择逻辑"）
- 高风险模块识别（如含新增/编辑/权限/金额等高业务价值模块）
- 全局扫描完成标志：已识别所有二级/三级标题对应的页面/模块

**子阶段 1b：逐页面走查** — 按全局视图中的页面顺序，对每个页面/模块按适用维度逐项检查。模块类型自动聚焦：

- 表单为主的页面 → 重点走查维度 1（控件规格）、2（按钮说明）、3（下拉数据源）、4（必填标注）、5（唯一性规则）、9（异常流程）
- 列表/管理页面 → 重点走查维度 2（按钮说明）、10（数据状态流转）、11（权限校验）
- 流程型页面 → 重点走查维度 6（功能覆盖）、7（边界条件）、9（异常流程）、10（数据状态流转）
- 仪表盘/报表页面 → 重点走查维度 12（接口与数据依赖）、13（验收标准）

**异常输入处理**：

- 文档不存在或路径错误 → 停止评审，告知"文件路径不存在，请确认后重试"，并列出可能的相似文件名
- 文档为空或内容不可解析 → 停止评审，告知"文档内容无法解析"
- 文档编码异常/乱码 → 停止评审，告知"文档编码异常，请确认文件为 UTF-8 编码后重试"
- 文档超长（>50 个页面/模块或 >10000 字）→ 建议切换 Quick 模式或使用 `--module` 分批走查
- 走查中途用户取消 → 保存已完成的走查笔记，告知"已保存 X/Y 个页面的走查结果，如需继续请重新触发"
- `--module` 匹配到多个同名模块 → 列出所有匹配项供用户选择，不自动取第一个

每完成一个页面的走查后，同步完成跨页面交叉比对（仅 Full 模式，维度 13 一致性），避免最后集中比对遗漏细节。跨页面比对产出追加到当前页面走查笔记的"跨页面比对标记"字段（Full 模式额外字段），格式为：

```markdown
- 跨页面比对标记：
  - 一致性风险：[与 XX 页面的 YY 操作行为/字段规格不一致]
  - 数据依赖风险：[XX 数据在 A 页面和 B 页面定义不一致或缺失]
```

**走查产出格式** — 每个页面走查完成后，产出如下结构化笔记：

```markdown
### [页面/模块名] 走查笔记

- 页面类型：[表单型 / 列表型 / 流程型 / 仪表盘型]
- 聚焦维度：[维度编号列表]
- 测试复杂度评估：[高/中/低]（[原因一句话]）
- ✖ 必须修正：
  1. [维度N] 具体问题描述
- ➕ 建议补充：
  1. [维度N] 具体问题描述
```

**Full 模式额外字段**（跨页面比对，仅在 Full 模式下追加）：

```markdown
- 跨页面比对标记：[与哪些页面有一致性/数据依赖关系]
```

如果页面走查结果为零问题，仍记录页面名和聚焦维度，避免遗漏页面。

**走查进度反馈** — 多页面文档（>3 个页面/模块）走查时，每完成一个页面向用户输出一行进度摘要：

> ✅ [页面名] 走查完成（2 ✖ / 3 ➕），进度 2/8

单页面文档或 Quick 模式不需要进度反馈。

**走查过半确认 [检查点 0.5]**（可选，仅 >5 页面时触发）— 走查完成半数页面后，向用户输出简短中期摘要并确认方向：

```markdown
**[检查点 0.5] 走查过半确认**（已完成 X/Y 页面）

当前发现：N ✖ / M ➕，主要测试风险集中在 [维度/页面名]。

是否继续？ / 需要调整聚焦维度？ / 终止评审？
```

用户无异议则继续走查剩余页面；用户要求调整则修改后续页面聚焦维度；用户终止则保存已完成走查笔记，进入检查点 1 生成部分报告。

**第二步：生成报告 [检查点 1]** — 按报告格式输出，每条问题标注分类标签。在存档前向用户展示审核结论摘要：

- 各严重级别问题数量汇总
- 执行摘要（一句话结论 + 测试可开始信号）
- ✖ 必须修正类问题列表摘要
- 询问用户：确认存档 / 需要补充评审 / 修改报告再存档

---

## 审阅深度

根据场景选择审阅深度：

**Quick 模式**：只检查 ✖ 必须修正类问题，聚焦维度 1（控件规格）、2（按钮说明）、4（必填标注）、5（唯一性规则）、8（验收标准）。适用于：紧急排期、小需求变更、需求变更的增量评审。

**Full 模式**：执行全部 15 维度完整走查。适用于：编写测试计划前、大版本 PRD、接手他人测试工作。

命令格式：

```
/qa-review <路径> --quick    # Quick 模式
/qa-review <路径>            # 默认 Full 模式
/qa-review <路径> --module <模块名>         # 指定模块走查
/qa-review <路径> --module <模块名> --quick # 指定模块 + Quick 模式
```

## 评审维度（15 维度）

### ★ 核心测试规格维度（Quick 模式必查）

---

### 1. 界面控件规格 ✖ 控件类型或长度限制缺失 / ➕ 格式要求未说明

> 没有控件规格，边界值测试无从下手。

逐一检查页面中出现的每一个输入控件（文本框、数字框、多行文本、日期选择等），确认以下项目是否有明确说明：

- **控件类型**：是文本输入、数字输入、日期选择、富文本还是其他？
- **长度限制**：最小/最大字符数或字节数是否明确？
- **格式限制**：是否有格式要求（如仅支持数字、字母、特定符号）？
- **默认值**：是否有默认值？默认值内容是什么？
- **只读/禁用规则**：在什么条件下控件不可编辑？

✖ 触发条件示例：

- 出现文本输入控件但无长度说明 → "姓名字段未说明最大长度，无法进行边界值测试"
- 出现数字控件但无范围说明 → "年龄字段未说明取值范围，无法确定边界值"

---

### 2. 按钮与操作说明 ✖ 按钮点击后行为未描述 / ➕ 按钮可见性/禁用条件缺失

> 没有按钮行为描述，无法设计操作路径测试用例。

逐一核查页面中出现的每一个按钮（包括图标按钮、链接按钮、操作列中的操作项），确认：

- **点击行为**：点击后触发什么操作？跳转到哪个页面？弹出什么弹窗？
- **成功反馈**：操作成功后有什么提示？页面如何变化？
- **失败反馈**：操作失败后有什么提示？
- **禁用/隐藏条件**：在什么状态下按钮不可点击或不显示？（如未选择数据时批量删除按钮是否禁用）
- **权限控制**：按钮是否有角色或权限限制？

✖ 触发条件示例：

- 出现"提交"按钮但无操作后行为说明 → "提交按钮行为未描述，无法测试提交后的页面状态"
- 出现"导出"按钮但无禁用条件 → "导出按钮缺少数据为空时的禁用/提示说明"

---

### 3. 下拉控件数据源 ✖ 下拉选项范围或来源未说明 / ➕ 空选项和默认选中未说明

> 下拉数据源不明确，无法准备测试数据，也无法测试数据联动。

逐一检查页面中出现的每一个下拉选择控件（Select、Dropdown、Radio、Checkbox 列表），确认：

- **选项范围**：是固定枚举值（需列出全部选项）还是动态数据（需说明数据来源接口或表）？
- **默认选中**：初始状态是否有默认选中项？是什么？
- **空选项**：是否允许不选？是否有"请选择"占位项？
- **联动规则**：选择某项后是否影响其他字段的选项范围？（如省市区三级联动）
- **编辑回显**：编辑状态下已有数据如何回显到下拉控件？

✖ 触发条件示例：

- 下拉控件"所属部门"无数据来源说明 → "部门下拉数据来源未说明，无法准备测试数据"
- 下拉控件"状态"无选项列表 → "状态下拉未列出全部枚举值，无法测试全路径"

---

### 4. 必填字段标注 ✖ 字段必填/非必填未说明 / ➕ 条件必填规则缺失

> 没有必填说明，空值异常测试无法设计，表单提交边界不清晰。

逐一检查页面中的每一个表单字段，确认：

- **必填标注**：是否明确标注哪些字段是必填（Required）、哪些是选填？
- **条件必填**：是否存在"当 A 字段选择 X 时，B 字段变为必填"的联动必填规则？
- **必填校验时机**：是提交时统一校验还是失焦时即时校验？
- **必填提示文案**：校验失败时显示什么提示文字？

✖ 触发条件示例：

- 表单中字段未标注必填/选填 → "备注字段是否必填未说明，空值提交测试无依据"
- 存在联动逻辑但无条件必填说明 → "选择'外部用户'时联系方式是否变为必填未说明"

---

### 5. 字段唯一性验证规则 ✖ 新增/编辑功能缺少唯一性说明 / ➕ 唯一性校验范围或提示文案缺失

> 无唯一性规则，重复数据测试无法进行，数据完整性无法验证。

针对所有包含新增或编辑功能的页面，检查：

- **唯一性字段**：哪些字段需要保证唯一（如用户名、编码、手机号）？
- **唯一性范围**：是全局唯一还是在特定范围内唯一（如同一部门下名称唯一）？
- **编辑时的唯一性**：编辑时是否需要排除自身再做唯一性校验（即同一条记录修改不触发冲突）？
- **校验时机**：是提交时校验还是输入时实时校验（如用户名实时检测是否重复）？
- **重复时提示文案**：校验失败时显示什么提示？

✖ 触发条件示例：

- 新增用户功能中"用户名"无唯一性规则说明 → "用户名是否唯一未说明，重复用户名测试无依据"
- 编辑功能无说明是否排除自身校验 → "编辑时唯一性校验逻辑不清晰，可能导致修改时误报重复"

---

### ★ 扩展测试维度（Full 模式追加，维度 6-15）

---

### 6. 功能覆盖完整性 ✖ 核心业务场景缺失 / ➕ 异常场景和边缘用户路径未描述

- 正常流程是否完整覆盖（主干路径）？
- 是否有角色/权限不同导致的功能差异场景？
- 审批/流程类功能是否覆盖全部流转节点？
- 并发操作场景是否有说明（如两人同时编辑同一条记录）？
- 批量操作场景是否覆盖（部分成功、全部失败、超量限制）？

---

### 7. 边界条件 ✖ 输入边界未定义 / ➕ 极限数据和超量策略未说明

- 数值类字段的最大/最小值边界是否明确？
- 列表类功能的最大数据量是否有说明（如最多添加 100 条）？
- 文件上传类功能的文件大小、格式、数量限制是否说明？
- 超出边界时的系统行为是截断、拒绝还是提示？
- 特殊字符（如 SQL 注入、XSS 字符、Emoji）是否有处理说明？

---

### 8. 测试验收标准 ✖ 功能无可量化的验收标准 / ➕ 性能和兼容性标准缺失 / 验收标准与正文不一致

- 每个功能点是否有明确的通过条件（Given/When/Then 或等价形式）？
- 关键业务指标是否有数字化验收标准（如查询响应时间 <2s）？
- 数据精度和格式的验收标准是否明确（如金额保留 2 位小数）？
- 兼容性要求（浏览器、操作系统、屏幕分辨率）是否说明？
- 性能基准（并发数、响应时间）是否说明？
- **【一致性交叉检查】验收标准与正文描述是否一致？逐条检查验收标准中的每一条，是否能在正文找到对应描述，若验收标准中的检查项在正文中无描述则为✖，若验收标准比正文描述更严格或更宽松也为✖**

---

### 9. 异常流程 ✖ 核心操作无异常处理说明 / ➕ 部分异常场景缺失

- 网络异常/超时时的用户反馈是否说明？
- 服务端返回错误时的页面行为是否说明？
- 操作中断（如关闭浏览器、断网后重连）的数据状态是否说明？
- 并发冲突（如库存超卖、重复提交）的处理机制是否说明？
- 长时间操作（如批量导入）的进度反馈和失败重试机制是否说明？

---

### 10. 数据状态与流转 ✖ 数据生命周期未定义 / ➕ 中间状态和回退路径未说明

- 数据的生命周期（创建→修改→归档/删除）是否完整？
- 状态机是否有说明（各状态名称、触发条件、允许的转换路径）？
- 状态回退或撤销是否有说明？
- 删除是物理删除还是软删除？软删除的数据是否可恢复？
- 状态变更是否触发通知或联动操作？

---

### 11. 权限与角色校验 ✖ 权限边界未定义 / ➕ 数据权限和操作权限未区分

- 各角色在该页面的可见范围是否说明（字段级、操作级）？
- 数据权限是否说明（只能看自己的数据、还是部门范围、还是全局）？
- 越权访问的处理方式是否说明（跳转到无权限页面 or 返回空数据）？
- 权限变更后对已打开页面的影响是否说明？

---

### 12. 接口与数据依赖 ✖ 页面数据来源不明 / ➕ 降级方案和缓存策略缺失

- 页面展示的数据字段是否有字段级定义（列表显示哪些列、详情展示哪些字段）？
- 数据获取方式是否明确（一次性加载/分页/下拉加载/实时搜索）？
- 依赖第三方或外部系统时，异常情况下的降级展示方案是否说明？
- 数据为空、加载中、加载失败时的页面展示是否说明？

---

### 13. 跨页面一致性 ✖ 同一字段在不同页面规格矛盾 / ➕ 操作确认方式不统一

- 同一字段在不同页面的类型、长度、格式定义是否一致？
- 同一操作（如删除）在不同页面的确认方式是否统一？
- 同一数据的展示格式在不同页面是否一致（如日期格式）？
- 权限控制逻辑在不同页面是否一致？
- 术语和命名是否全文统一？

---

### 14. 错误提示规范 ✖ 核心操作无错误提示定义 / ➕ 提示方式不统一、文案模糊

- 表单校验失败是否有具体提示文案？
- 操作失败是否告知具体原因而非只显示"系统错误"？
- 错误提示的展示方式是否统一（Toast/弹窗/字段下红字）？
- 前端本地校验（如文件超限）是否有独立的提示文案定义？

---

### 15. AI智能体交互 ✖ AI引导策略缺失 / ➕ 意图识别兜底和话术规范缺失

> 当被测系统包含AI智能体/聊天机器人时，必须检查以下内容。

**15.1 AI引导策略（输入不充分时）**
- 当用户输入信息不足时，AI是否有明确的引导话术？（如"请提供姓名或手机尾号"）
- AI引导话术是否固定？有无场景化差异？
- 当多轮对话时，AI是否能结合前文继续处理？还是必须完整输入？
- 超时机制：用户长时间不补充信息，AI是否自动结束会话？

**15.2 AI意图识别失败处理**
- AI无法识别用户意图时的回复话术是什么？是否有兜底文案？
- 无关内容（如"今天天气不错"）和有效但无法处理的意图如何区分？
- 意图识别失败的降级处理：是否转人工？是否有推荐操作？

**15.3 意图识别完整性**
- 意图识别表格是否覆盖所有支持的意图？（如入住、退房、查询、续住等）
- 每个意图的触发关键词是否明确？有无歧义风险？
- 相似意图是否有区分策略？（如"续住"和"延期"的区分）

**15.4 意图识别准确性与边界**
- 是否有误识别风险？（如"退房"被误识别为入住）
- 当输入同时匹配多个意图时，系统如何处理？
- 意图识别是否支持否定表达？（如"我不是要入住"）

✖ 触发条件示例：
- AI输入"客人办理入住"无引导策略说明 → "AI缺少信息不完整时的引导策略"
- AI输入"今天天气不错"无兜底回复 → "AI无法识别意图时无降级处理说明"

## Common Mistakes（QA 视角高频失误）

1. **只测主干路径，不测边界** → 长度上限、空值、特殊字符均未覆盖
2. **下拉选项准备不足** → 测试数据构造失败，阻塞用例执行
3. **忽略唯一性规则** → 重复数据写入，数据完整性问题上线后才暴露
4. **必填判断靠猜** → 空值测试用例设计出错，或测出误报问题
5. **按钮行为不明** → 点击后不知道期望结果，无法判断是 Bug 还是需求如此
6. **权限场景遗漏** → 越权访问问题在测试阶段未覆盖
7. **异常流程不验收** → 网络超时、并发冲突等场景上线后集中爆发
8. **跨页面一致性未核对** → 同一字段在多处定义不同，测试结果互相矛盾

## HTML 报告生成规则

评审完成后，除 Markdown 报告外，**必须**同时生成 HTML 格式报告。

**模板位置**：与 SKILL.md 同目录的 `report-template.html`

**生成步骤**：

1. 读取 `report-template.html` 模板文件
2. 替换 `{{变量名}}` 占位符为评审数据（变量对照表见下方）
3. 循环区块：模板中 `<!-- BEGIN:sectionName -->` 至 `<!-- END:sectionName -->` 之间的内容为循环单元，每个页面/每个问题复制一份，替换其中的占位符
4. 空值处理：若某页面无问题，输出 `<div class="no-issues">无问题</div>`
5. 写入文件，编码 UTF-8

**占位符对照表**：

| 占位符                 | 对应数据                 | 来源                     |
| ---------------------- | ------------------------ | ------------------------ |
| `{{docName}}`          | 需求文档名称             | 报告基本信息             |
| `{{docVersion}}`       | 文档版本                 | 报告基本信息             |
| `{{reviewTime}}`       | 评审时间                 | 报告基本信息             |
| `{{reviewScope}}`      | 评审范围                 | 报告基本信息             |
| `{{reviewDepth}}`      | `quick` 或 `full`        | 审阅深度（控制徽章样式） |
| `{{execSummary}}`      | 执行摘要一句话           | 报告执行摘要             |
| `{{verdictClass}}`     | `pass` / `warn` / `fail` | 评审结论                 |
| `{{verdictText}}`      | 结论文字                 | 评审结论                 |
| `{{fixCount}}`         | ✖ 修正问题总数           | 问题汇总                 |
| `{{suggestCount}}`     | ➕ 补充问题总数          | 问题汇总                 |
| `{{summaryTableRows}}` | 汇总表行 HTML            | 页面维度汇总             |
| `{{pageName}}`         | 页面/模块名              | 循环：问题详情按页面分组 |
| `{{issueTitle}}`       | 问题标题                 | 循环：每个问题           |
| `{{issueLocation}}`    | 文档章节/页面位置        | 循环：每个问题           |
| `{{issueDimension}}`   | 所属维度                 | 循环：每个问题（新增）   |
| `{{issueDesc}}`        | 问题描述                 | 循环：每个问题           |
| `{{issueImpact}}`      | 对测试的影响             | 循环：每个问题           |
| `{{issueSuggestion}}`  | 改进建议                 | 循环：每个问题           |

> 跨页面比对结果不出现在 HTML 报告中（已融入各页面问题描述），仅在 Markdown 报告的走查笔记中保留。

## 存档规范

评审完成后，**必须**将评审报告自动存档到 skill 子目录 `docs/` 中。

**存档路径格式**：

```
docs/{文档名称}-QA评审报告-{YYYYMMDDHHMMSS}.md
docs/{文档名称}-QA评审报告-{YYYYMMDDHHMMSS}.html
```

**示例**：

- `docs/用户管理模块-QA评审报告-20260523101010.md`
- `docs/用户管理模块-QA评审报告-20260523101010.html`

**同名冲突处理**：如同文档同一秒内已有存档（极少见），文件名末尾追加 `-2`、`-3` 后缀。如果用户对已有报告不满意需重新评审，询问用户是覆盖旧报告还是保留旧报告（追加后缀）。

## 使用方法

### 命令格式

| 模式         | 命令                                            | 说明                             |
| ------------ | ----------------------------------------------- | -------------------------------- |
| Full         | `/qa-review <路径>`                             | 默认，全部 15 维度               |
| Quick        | `/qa-review <路径> --quick`                     | 只查 ✖ 修正，5 核心测试维度      |
| 模块         | `/qa-review <路径> --module <模块名>`           | 指定模块走查                     |
| 模块 + Quick | `/qa-review <路径> --module <模块名> --quick`   | 指定模块 + Quick 模式            |

### --module 模式差异

与标准流程的区别仅在检查点 0：展示匹配到的模块内容摘要，而非全文档页面列表。如果未找到指定模块，列出文档中所有可用模块供用户选择。模块识别规则：按文档中的二级/三级标题匹配，支持部分匹配（如"用户"可匹配"用户管理"）。

## 自然语言触发规则

当用户未使用 `/qa-review` 命令格式，而是用自然语言描述评审意图时（如"帮我看看这个需求文档，我要开始写测试用例了"、"测试评审一下这个 PRD"），应：

1. **优先确认文档路径** — 如果用户没有提供文件路径，主动询问："请提供需求文档的文件路径"，不可凭空评审
2. **推断审阅深度** — 根据用户语气推断：紧急/快速 → Quick 模式；全面/仔细 → Full 模式；推断不准时默认 Full 模式并告知用户
3. **转为标准流程** — 获取路径后，按标准执行流程开始（检查点 0 → 走查 → 检查点 1）

示例对话：

```
用户: "产品给了一个新需求文档，我准备开始写测试用例，帮我先评审一下"
AI:   收到，我会从测试视角对您的需求文档进行全面评审（Full 模式，15 维度）。
      请提供需求文档的文件路径，例如：requirements/prd/xxx.md
      [获取路径后进入检查点 0]
```

## 注意事项

1. 此 skill 从测试视角评审需求，不替代 PM 和开发的需求评审
2. 评审结果基于预设规则，可能存在遗漏，建议结合实际业务场景判断
3. Quick 模式聚焦 5 个核心维度，可能遗漏扩展维度问题，建议正式评审使用 Full 模式
4. 评审报告必须存档到 `docs/` 子目录，便于追溯版本间的需求变化
5. 发现 ✖ 必须修正问题时，建议在评审会议前同步给 PM 和开发，避免会议时间浪费
