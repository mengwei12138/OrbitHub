# 产品需求文档设计规范

## 执行约束（强制要求，AI不得跳过任何条款）

**警告**：如果不执行本skill中的要求，将导致以下问题：
- 生成的PRD缺少关键规格说明，开发人员无法准确实现
- 缺少字段规格导致测试用例无法设计边界值测试
- 缺少下拉数据源说明导致测试数据准备失败
- 缺少按钮行为说明导致测试路径无法覆盖
- 缺少控件清单和控件绑定规则导致前端实现遗漏页面元素
- 缺少状态流转和弹窗字段导致交互实现与测试验收口径不一致
- 缺少验收标准导致上线验收无依据

**因此，AI必须严格执行每一项要求，任何跳过行为都视为不合格输出。**

**强制执行条款**：

1. **每一条要求都必须执行**，不得跳过、简化或省略任何条款
2. 如遇特殊情况无法执行某条款，必须先询问用户获得确认
3. 生成PRD时，**必须逐条检查测试点覆盖清单**，确保每项都已覆盖
4. 对于"每个"、"全部"、"逐个"等词汇描述的要求，**必须100%覆盖**，不得选择性执行
5. 文档中未明确说明的内容（如字段规格、按钮行为、数据源），**必须主动补充完整**，不得留空
6. 异常流程**必须明确清晰完整地进行全量输出**，**严禁简化或省略任何异常场景**
7. 每个页面、弹窗、抽屉、步骤页和独立交互区**必须**输出控件清单表、控件绑定业务规则表和必要的控件统计
8. 如功能存在状态、Tab、分类、枚举、审批节点、任务进度或异步处理，**必须**输出状态/选项枚举表和状态流转说明
9. 如功能存在弹窗、表单、抽屉或批量操作确认框，**必须**输出弹窗/表单字段表，并说明每个按钮的规则
10. 完成PRD后**必须**检查控件数量、规则绑定、状态枚举、弹窗字段与验收标准是否自洽

## 文档内容要求

### 基本要求

- 按本规范生成PRD文档，不提供接口需求和数据模块
- 验收标准**必须**按用例设计形式完善
- 遇到”可能”、”等”等模糊描述时，**必须**先询问用户确认，不可自行推断

### 文档结构

PRD文档**必须**包含以下8个章节，**禁止跳过任何章节**：

| 序号 | 章节名称 | 内容要求 |
|------|---------|---------|
| 1 | 需求概述 | 介绍模块功能、使用场景、进入路径（从哪个页面跳转） |
| 2 | 名词解释 | 解释有歧义的词汇，无关名词不在此版本说明 |
| 3 | 流程图 | **必须**包含模块流程图和功能流程图 |
| 4 | 角色说明 | 说明系统角色及权限，**必须**绘制跨职能泳道图 |
| 5 | 功能清单 | 表格形式，包含版本、模块、功能点、描述、优先级；如页面较多，必须补充各页面控件清单汇总和控件统计 |
| 6 | 详情功能设计 | 对每个功能点**必须**按结构化要求详细产出（见下方说明），覆盖页面、控件、规则、状态、弹窗、异常 |
| 7 | 非功能性设计 | 数据需求、性能需求、安全需求 |
| 8 | 验收标准 | 表格形式，包含所有逻辑、交互、按钮、异常的验收条件 |

### 详情功能设计要求

**必须**按以下6个部分详细产出：

| 部分 | 要求说明 |
|------|---------|
| 功能概述 | 说明功能用途、业务价值 |
| 前置条件 | 角色、权限、状态要求 |
| 后置条件 | 数据变化、页面跳转 |
| 界面交互 | 包含页面展示、按钮逻辑、元素说明 |
| 业务流程 | 业务流程图+文字描述，包含异常流程 |
| 异常与分支流程 | **必须**全量输出（见测试点覆盖清单） |
| 结构化支撑表 | 页面与跳转关系、控件清单、控件绑定规则、状态枚举、弹窗字段、控件统计 |

**界面交互要求**：
- **页面展示**：**必须**输出所有页面统一的原型界面线框图，**禁止跳过任何页面**，大小样式一致
- **按钮逻辑**：**必须**说明**每个按钮（无例外）**的点击行为、跳转逻辑、逻辑处理
- **元素说明**：**必须**列出**每个元素（无例外）**的字段名称、控件类型、是否必填、默认值、长度、规则等

### 结构化输出要求

**强制要求**：每个页面或弹窗必须按以下结构补充支撑表。若某项不适用，必须写明"不适用，原因：XXX"，不得直接省略。

| 结构化块 | 输出要求 |
|----------|---------|
| 页面与跳转关系 | 列出页面清单、入口、返回路径、页面内跳转、外部模块跳转和跳转条件 |
| 控件清单表 | 列出页面/弹窗内全部控件，包含按钮、输入框、下拉、表格列、Tab、指标、标签、分页器、开关、复选框、单选、多选、提示区和空状态入口 |
| 控件统计 | 按页面汇总控件数量和主要类型分布；控件总数必须等于各控件清单表明细数量 |
| 控件绑定业务规则表 | 将每条业务规则绑定到具体控件，说明触发条件、处理逻辑和前端反馈 |
| 字段规格表 | 对每个表单字段说明控件类型、必填、默认值、长度、格式、校验、错误提示和回显规则 |
| 按钮规格表 | 对每个按钮说明点击行为、成功反馈、失败反馈、禁用/隐藏条件、权限控制和防重复规则 |
| 下拉/枚举数据源表 | 对每个下拉、单选、多选、Tab、分类、状态筛选列出全部选项、默认值、空选项、来源和联动规则 |
| 状态枚举表 | 对每个业务状态、任务状态、按钮状态、记录状态说明枚举值、绑定控件、操作可用性、进入条件和退出条件 |
| 状态流转说明 | 用流程图或表格说明从初始状态到终态的完整状态流转，包含成功、失败、取消、重试、超时和回滚分支 |
| Tab/分类表 | 如页面包含Tab或分类，列出Tab/分类名称、计数规则、默认页、筛选条件、优先级和冲突处理 |
| 弹窗/表单字段表 | 如存在弹窗、抽屉或确认框，逐个列出字段、控件类型、必填、默认值、校验规则、错误提示和按钮行为 |
| 异常与分支流程 | 覆盖网络、权限、校验、并发、异步任务、第三方失败、用户取消、重复提交、数据为空和数据过期等场景 |

**控件清单表模板**：

| 序号 | 控件名 | 控件类型 | 所属区域 | 说明 |
|------|--------|----------|----------|------|
| 1 | 示例控件 | 按钮/输入框/表格列/Tab/分页器等 | 页面区域或弹窗名称 | 控件用途、展示条件、交互说明 |

**控件绑定业务规则表模板**：

| 规则 | 绑定控件 | 触发条件 | 处理逻辑 | 反馈 |
|------|----------|----------|----------|------|
| 示例规则 | 示例控件 | 用户操作或系统状态 | 校验、保存、跳转、异步任务或联动逻辑 | Toast/弹窗/字段错误/状态变化 |

**状态枚举表模板**：

| 状态 | 枚举值 | 绑定控件 | 操作可用性 | 进入条件 | 退出条件 |
|------|--------|----------|------------|----------|----------|
| 初始状态 | `initial` | 示例按钮/列表行 | 可点击/不可点击/只读 | 页面加载或记录创建 | 用户操作、任务完成或异常 |

**弹窗/表单字段表模板**：

| 字段 | 控件类型 | 是否必填 | 默认值 | 校验规则 | 错误提示 |
|------|----------|----------|--------|----------|----------|
| 示例字段 | 输入框/下拉/复选框 | 必填/选填/条件必填 | 默认值或空 | 长度、格式、唯一性、联动规则 | 具体错误文案 |

**统计自洽要求**：
- 控件清单表中的控件数量必须与控件统计一致
- 控件绑定业务规则表中的"绑定控件"必须能在控件清单表中找到
- 状态枚举表中的"绑定控件"必须能在控件清单表或按钮规格表中找到
- 弹窗/表单字段表中的按钮必须在按钮规格表或控件绑定业务规则表中说明
- Tab、分类、枚举、状态筛选必须列出全部选项，不得使用"等"或"包括但不限于"

### 字段规格要求

**强制要求**：每个表单字段**必须**逐个说明以下规格，**禁止遗漏任何字段**：

| 规格项 | 必须说明的内容 |
|--------|--------------|
| 控件类型 | 文本/数字/日期/下拉/单选/多选/密码等 |
| 长度限制 | 最小/最大字符数或字节数 |
| 格式限制 | 正则表达式或格式说明 |
| 默认值 | 固定值或计算规则 |
| 只读/禁用 | 在什么条件下字段不可编辑 |
| 是否必填 | 必填/选填，条件必填时说明联动规则 |
| 校验时机 | 提交时校验/失焦时即时校验 |
| 唯一性 | 是否需要唯一性校验，校验范围 |
| 错误提示 | 校验失败时的具体提示文案 |

### 下拉控件数据源要求

**强制要求**：每个下拉控件**必须**说明以下内容，**禁止跳过**：

| 规格项 | 必须说明的内容 |
|--------|--------------|
| 选项来源 | 固定枚举值（列出全部）/ 动态数据来源 |
| 默认选中 | 初始状态的默认选中项 |
| 空选项 | 是否允许不选，是否有"请选择"占位项 |
| 联动规则 | 选择后影响哪些字段的选项范围 |
| 编辑回显 | 编辑状态下已有数据如何回显 |

### 按钮规格要求

**强制要求**：每个按钮**必须**说明以下内容，**禁止跳过**：

| 规格项 | 必须说明的内容 |
|--------|--------------|
| 点击行为 | 点击后触发什么操作（跳转/弹窗/提交/刷新） |
| 成功反馈 | 操作成功后的提示内容和页面变化 |
| 失败反馈 | 操作失败时的具体错误提示文案 |
| 禁用/隐藏条件 | 什么状态下按钮不可点击或不显示 |
| 权限控制 | 按钮是否有角色或权限限制 |

## 线框图规范

- **必须**使用ASCII/文本格式
- 大小格式**必须**一致
- 同一界面**禁止**出现歧义按钮或功能重复的按钮
- **必须**整洁无逻辑错误
- 可使用颜色突出重点功能

## 需求清晰度检测模块

### 检测要点

- 需求描述：是否清晰、明确，无模糊、歧义的表述，是否明确"做什么""不做什么"；
- 场景定义：是否明确AI功能的使用场景、触发条件、用户操作路径，场景边界清晰；
- 指标明确：功能指标、性能指标、体验指标是否量化（如"提升推荐准确率"需明确提升至多少，"优化响应速度"需明确具体耗时）；
- 边界清晰：明确需求的范围，不包含无关功能，不出现"模糊化"需求（如"优化AI体验""提升算法效果"，无具体量化标准）。
- 线框图逻辑：基于需求文档的内容，检查线框图的合理性。
- **AI智能体交互**（如功能包含AI聊天）：**必须**检测引导策略是否明确、意图识别是否完整、兜底话术是否定义、超时机制是否说明。

### 检测方法

AI通过自然语言处理技术，解析需求说明书，自动识别模糊、歧义表述，校验场景定义、指标量化、边界划分的清晰度，判断需求是否具备可执行性。

### 判断标准

合格：需求描述清晰、场景明确、指标量化、边界清晰，无模糊歧义，研发人员可根据需求直接开展工作；

不合格：需求模糊、歧义较多，场景边界不清晰，指标未量化，无法指导研发工作。如果出现不合格的需求点，请帮我修复，但每修复一条都需要进行询问确认。

## 测试点覆盖清单

**强制要求**：编写需求文档时，**必须覆盖以下所有测试点场景，100%全覆盖，0%遗漏率**。

如有任何测试点因特殊原因无法覆盖，**必须**：
1. 在PRD中明确标注"该测试点不适用于本功能，原因：XXX"
2. 获得用户确认后方可跳过
3. **严禁未经确认自行跳过任何测试点**

### 1. 列表功能测试点

| 测试类型 | 必须覆盖的测试场景 |
|----------|-------------------|
| 正常展示 | 列表有数据时正常展示，数据卡片/列表项完整显示 |
| 空数据 | 列表无数据时显示缺省页面，显示对应提示文案和操作入口 |
| 分页加载 | 每页10条分页，上滑加载更多，加载完毕显示"没有更多数据了" |
| 下拉刷新 | 顶部下拉刷新整个列表，数据更新成功 |
| 加载失败 | 网络异常/超时/服务器错误时显示错误页，提供重试按钮 |

### 2. 表单验证测试点

| 测试类型 | 必须覆盖的测试场景 |
|----------|-------------------|
| 必填校验 | 必填字段为空时提交按钮禁用，或提交后提示"XXX不能为空" |
| 长度边界 | 每个文本输入框需明确最小/最大长度，并测试：最小-1字符、最小字符、正常、最大字符、最大+1字符 |
| 格式校验 | 数字字段需校验整数/小数位数、负数；日期字段需校验格式；特殊格式字段需明确格式要求 |
| 重复校验 | 唯一性字段（如名称、编码）需校验重复，提示"XXX已存在" |
| 安全过滤 | 用户输入内容需考虑XSS注入风险，明确是否过滤特殊字符（`<script>`等） |

### 3. 按钮交互测试点

| 测试类型 | 必须覆盖的测试场景 |
|----------|-------------------|
| 确定按钮 | 点击执行操作，操作成功后关闭对话框/跳转页面 |
| 取消按钮 | 点击放弃当前操作，关闭对话框，表单数据不保存 |
| 关闭按钮[X] | 点击等同于取消，关闭对话框，数据不保存 |
| 返回按钮 | 点击返回上一页，需明确：停留当前页/返回列表/其他行为 |
| 防重复点击 | 按钮点击后禁用防抖，防止重复提交 |

### 4. 状态切换测试点

| 测试类型 | 必须覆盖的测试场景 |
|----------|-------------------|
| 状态反转 | 上下架状态点击直接切换，无需确认 |
| 批量操作 | 多选后批量操作需弹出确认框，显示选中数量 |
| 状态回显 | 编辑页面需正确回显所有字段的当前值 |

### 5. 关联影响测试点

删除/修改操作必须明确对关联数据的影响：

| 测试场景 | 必须明确的内容 |
|----------|---------------|
| 删除操作 | 删除后关联表数据如何处理（如供应商关联、历史订单是否受影响） |
| 状态变更 | 状态变更对其他业务的影响（如上架商品可被选择、下架商品不可被选择） |
| 数据引用 | 修改某数据后，依赖该数据的其他数据如何处理 |

### 6. 异常流程测试点

| 异常类型 | 必须覆盖的测试场景 |
|----------|-------------------|
| 网络断开 | 提示"网络异常，请检查网络连接"，显示重试按钮 |
| 请求超时 | 提示"请求超时，请稍候重试"，显示重试按钮 |
| 接口返回异常 | 后端返回错误码时，提示具体错误信息 |
| 服务器内部错误 | 提示"系统异常，请稍候重试"，显示重试按钮 |
| Token失效 | 跳转登录页面 |
| 重复提交 | 按钮防抖，不重复调用接口 |
| 并发冲突 | 两人同时操作同一数据时的处理机制 |
| 操作中断 | 关闭浏览器/断网后重连，数据状态是否保留 |
| 长时间操作 | 批量导入/导出时的进度反馈和失败重试机制 |

### 7. 性能指标测试点

必须量化以下性能指标：

| 操作类型 | 性能要求 |
|----------|----------|
| 列表首次加载 | ≤1秒 |
| 分页加载 | ≤500ms |
| 表单提交（含图片） | ≤5秒 |
| 条码扫描识别 | ≤2秒 |

### 8. 安全测试点

| 测试类型 | 必须覆盖的测试场景 |
|----------|-------------------|
| XSS注入 | 输入`<script>alert(1)</script>`等脚本，验证是否过滤或执行 |
| SQL注入 | 输入`' OR 1=1 --`等SQL片段，验证是否过滤 |
| 越权访问 | 用户A创建的数据，用户B不能访问/修改/删除 |
| 文件上传 | 上传不支持格式（如GIF）、超过大小限制、包含恶意代码的文件 |
| 特殊字符 | Emoji、表情符号、SQL注入字符、XSS字符等的处理方式 |

### 9. 操作审计测试点

如业务有审计要求，需明确：

| 测试场景 | 需明确的日志内容 |
|----------|---------------|
| 操作记录 | 哪些操作需要记录日志（增/删/改） |
| 日志内容 | 日志需记录：操作人、操作时间、操作类型、操作对象、操作前后数据 |
| 验证方式 | 日志可通过接口查询还是需直接查数据库 |

### 10. 预置数据定义

如有预置数据（如预置类别、预置配置），必须明确：

| 需定义的内容 | 示例 |
|-------------|------|
| 预置数据列表 | 预置11个商品类别及其名称 |
| 预置数据标识 | 预置数据与用户创建数据的区分标识 |
| 预置数据颜色 | 颜色定义，如：电子产品#1890FF、文具用品#52C41A |
| 自定义数据颜色 | 用户创建数据时颜色随机分配规则 |

### 11. 权限与角色校验测试点

**强制要求**：如功能涉及权限控制，**必须**说明以下测试场景：

| 测试场景 | 必须明确的内容 |
|----------|---------------|
| 字段可见性 | 各角色在页面中的可见范围（字段级权限） |
| 操作权限 | 各角色的操作权限（哪些按钮可见/可点击） |
| 数据权限 | 只能看自己的数据/部门范围/全局 |
| 越权处理 | 越权访问时的处理方式（跳转无权限页/返回空数据） |
| 权限变更 | 权限变更对已打开页面的影响 |

### 12. 跨页面一致性测试点

**强制要求**：PRD中**必须**包含以下一致性检查：

| 检查项 | 必须明确的内容 |
|--------|--------------|
| 术语统一 | 全文使用统一的术语和命名规范 |
| 字段规格一致 | 同一字段在不同页面的类型、长度、格式定义一致 |
| 操作方式统一 | 同一操作（如删除）在不同页面的确认方式一致 |
| 数据格式统一 | 日期等数据格式在全文统一 |
| 权限逻辑统一 | 权限控制逻辑在不同页面一致 |

### 13. 错误提示规范测试点

**强制要求**：PRD中**必须**明确以下错误提示规范：

| 测试场景 | 必须明确的内容 |
|----------|---------------|
| 校验提示文案 | 表单校验失败的具体提示文案 |
| 失败原因 | 操作失败是否告知具体原因（非仅"系统错误"） |
| 提示方式 | 错误提示的展示方式（Toast/弹窗/字段下红字） |
| 本地校验 | 前端本地校验（如文件超限）的独立提示文案 |

### 14. AI智能体交互测试点（如功能包含AI聊天）

**强制要求**：如功能包含AI聊天，**必须**说明以下测试场景：

**AI引导策略（14.1）**
| 测试场景 | 必须明确的内容 |
|----------|---------------|
| 引导话术 | 输入不充分时的具体引导话术（列出每种场景的话术） |
| 多轮对话 | 是否能结合前文继续处理 |
| 超时机制 | 会话超时时间、超时后的处理 |

**AI意图识别失败处理（14.2）**
| 测试场景 | 必须明确的内容 |
|----------|---------------|
| 兜底话术 | 无法识别意图时的回复话术 |
| 无关内容处理 | 无关内容和有效但无法处理的意图如何区分 |
| 降级处理 | 是否转人工、推荐操作 |

**意图识别完整性（14.3）**
| 测试场景 | 必须明确的内容 |
|----------|---------------|
| 意图列表 | 支持的意图列表（如入住、退房、查询、续住） |
| 触发关键词 | 每个意图的触发关键词 |
| 相似意图区分 | 相似意图的区分策略 |

**意图识别准确性（14.4）**
| 测试场景 | 必须明确的内容 |
|----------|---------------|
| 误识别风险 | 误识别风险及处理方式 |
| 多意图处理 | 同时匹配多个意图时的处理方式 |
| 否定表达 | 是否支持否定表达（如"我不是要入住"） |

### 15. 结构化PRD完整性测试点

**强制要求**：PRD中**必须**包含以下结构化检查，确保文档可被产品、设计、研发、测试直接拆解使用：

| 检查项 | 必须明确的内容 |
|--------|---------------|
| 控件全量覆盖 | 每个页面、弹窗、抽屉、表单、Tab页的控件是否全量列出 |
| 控件统计一致 | 每页控件数、分类型控件数、全模块控件总数是否与控件清单明细一致 |
| 规则绑定完整 | 每个按钮、开关、复选框、下拉、Tab、行内操作是否都有绑定业务规则 |
| 状态流转完整 | 每个状态是否有枚举值、进入条件、退出条件、操作可用性和异常分支 |
| 弹窗字段完整 | 每个弹窗/表单是否列出字段、按钮、默认值、校验规则、错误提示和关闭行为 |
| Tab/分类/枚举完整 | Tab、分类、状态、筛选项、单选/多选项是否列出全部选项和默认值 |
| 跨页面一致性 | 同名字段、同类按钮、分页、搜索、删除、启停、提交、取消等操作口径是否一致 |

### 测试点覆盖检查清单

编写完需求文档后，对照以下清单**逐条检查**，**确保每项都已覆盖**，**禁止跳过任何一项**：

- □ 列表功能：正常/空数据/分页/刷新/加载失败
- □ 表单验证：必填/长度边界(4个值)/格式/重复/安全过滤
- □ 字段规格：**每个字段**的控件类型/长度/格式/默认值/只读禁用/校验时机/唯一性/错误提示
- □ 下拉数据源：选项来源/默认选中/空选项/联动规则/编辑回显
- □ 按钮交互：确定/取消/关闭[X]/返回/防重复/成功反馈/失败反馈/禁用条件
- □ 状态切换：直接反转/需确认/批量操作/回显
- □ 关联影响：删除影响/状态变更影响/数据引用处理
- □ 异常流程：网络断开/请求超时/接口异常/服务器错误/Token失效/重复提交/**并发冲突**/**操作中断**/**长时间操作**
- □ 性能指标：列表加载/分页/表单提交/条码识别 均有量化指标
- □ 安全测试：XSS注入/SQL注入/越权访问/文件上传安全/**特殊字符处理**
- □ 操作审计：操作记录范围/日志内容/验证方式
- □ 预置数据：数据列表/标识/颜色/自定义规则
- □ **权限校验**：字段可见性/操作权限/数据权限/越权处理/权限变更影响
- □ **跨页面一致性**：术语统一/字段规格一致/操作方式统一/数据格式统一
- □ **错误提示规范**：校验提示文案/失败原因/提示方式/本地校验
- □ **AI智能体交互**（如包含AI）：引导策略/兜底话术/意图列表/触发关键词/超时机制/误识别处理
- □ **结构化PRD完整性**：控件全量列出/控件数与统计一致/每个按钮有绑定规则/每个状态有进入和退出条件/每个弹窗有字段和按钮规则/Tab、分类、枚举列出全部选项/跨页面同名字段和同类操作一致

## 使用方法

当用户需要进行产品需求设计时，加载此技能：

```
/skill prd-design
```

加载后，**必须**按照本规范输出PRD文档，包含8个标准章节（见"文档结构"），并在完成后**逐条检查**测试点覆盖清单（见"测试点覆盖检查清单"）。

**强制要求**：生成PRD时，**必须逐条执行**上述所有要求，**禁止跳过任何一项**。如遇特殊情况无法执行某条款，必须先询问用户获得确认。
